# Coding Conventions

**Analysis Date:** 2026-03-07

## Naming Patterns

**Files:**
- Screen components: PascalCase with `Screen` suffix (e.g., `WelcomeScreen.tsx`, `DashboardScreen.tsx`)
- UI components: kebab-case for shadcn/ui primitives (e.g., `button.tsx`, `alert-dialog.tsx`, `input-otp.tsx`)
- Custom components: PascalCase (e.g., `AnimatedGradient.tsx`, `ImageWithFallback.tsx`)
- Hooks: camelCase with `use-` prefix (e.g., `use-mobile.ts`)
- Utility files: camelCase (e.g., `utils.ts`)

**Functions:**
- Use `function` declarations for React components, not arrow functions: `export function WelcomeScreen() {}`
- Exception: shadcn/ui components also use `function` declarations but without `export` keyword on the function itself; they export via a named export at the bottom
- Helper functions inside components use arrow functions: `const getGreeting = () => {}`
- Event handlers use `handle` prefix: `handleStart`, `handleContinue`, `handleSearch`, `handleClusterClick`

**Variables:**
- camelCase for all variables and state: `timeLeft`, `isRecording`, `connectMode`, `selectedCluster`
- State interfaces use PascalCase: `WidgetState`, `Cluster`, `Connection`, `StickyNote`
- Constants use UPPER_SNAKE_CASE: `ITEM_TYPE`, `MOBILE_BREAKPOINT`, `ERROR_IMG_SRC`

**Types:**
- Use TypeScript `interface` for component props and data shapes (e.g., `interface Cluster {}`, `interface StickyNote {}`)
- Inline prop types for simple component props using destructuring: `function WordCloud({ word, delay }: { word: string; delay: number })`
- Use `React.ComponentProps<"element">` pattern for extending HTML element props in UI components

## Import Organization

**Order:**
1. React and React hooks (`import { useState, useEffect } from "react"`)
2. Third-party libraries (`react-router`, `motion/react`, `lucide-react`, `react-dnd`)
3. Internal UI components (`../components/ui/button`, `../components/ui/card`)
4. Internal custom components (`../components/AnimatedGradient`)
5. Assets (`figma:asset/...`)

**Path Aliases:**
- `@` maps to `./src` (configured in `vite.config.ts`)
- However, all imports in screen files use relative paths (`../components/ui/button`) rather than the alias
- The `figma:asset/` prefix is used for image imports: `import cloudImage from 'figma:asset/3c7af6a14225d1ee2a77d186872f69245c52483a.png'`

**Pattern:** Use relative imports from screen/component files. Use `@/` alias only if importing across distant directory boundaries.

## Component Patterns

**Screen Components:**
- Each screen is a single exported function component in its own file under `src/app/screens/`
- All state, helpers, and sub-components live in the same file (no separate hooks or utility extraction)
- Use named exports: `export function WelcomeScreen() {}`
- Navigation via `useNavigate()` from `react-router`

**UI Components (shadcn/ui):**
- Follow the shadcn/ui pattern: function declarations, `cn()` utility for className merging, `data-slot` attributes
- Accept `className` and spread remaining props: `function Input({ className, type, ...props }: React.ComponentProps<"input">)`
- Use `class-variance-authority` (`cva`) for variant-based styling (see `src/app/components/ui/button.tsx`)
- Export at bottom of file: `export { Button, buttonVariants }`

**Animation Pattern:**
- Use `motion` from `motion/react` (Framer Motion) for all animations
- Standard entry animation: `initial={{ opacity: 0, y: 20 }} animate={{ opacity: 1, y: 0 }}`
- Stagger with incremental `delay`: `transition={{ delay: 0.1 }}`, `transition={{ delay: 0.2 }}`, etc.
- Use `AnimatePresence` for conditional rendering animations

## Styling Approach

**Primary:** Tailwind CSS v4 utility classes via `className`

**Inline `style` for colors:** The codebase uses inline `style` objects for all color values rather than Tailwind color utilities:
```tsx
style={{ color: '#0D0D0D' }}
style={{ backgroundColor: '#F7F5F0' }}
style={{ borderColor: '#E8E5E0' }}
```

**Design System Colors (use these exact hex values):**
- Primary text: `#0D0D0D`
- Secondary text: `#6B6B6B`
- Muted text: `#A8A49E`
- Background: `#F7F5F0`
- Card background: `#FFFFFF`
- Cream: `#F2EFE9`
- Border: `#E8E5E0`
- Yellow (CTA): `#F5E642`
- Peach: `#F5C4A1`
- Sage: `#C8D5B0`
- Blush: `#F0D5D0`
- Dark: `#1A1A1A`

**Font Families (set via className, not style):**
- Headings (large): `font-['Playfair_Display']`
- Headings (medium): `font-['Lora']`
- Body text: `font-['DM_Sans'] font-light`
- Labels/buttons: `font-['Outfit'] font-semibold text-xs tracking-[0.08em] uppercase`
- Accents/quotes: `font-['Cormorant_Garamond'] italic font-light`

**Button Styling Pattern:**
```tsx
<Button
  className="rounded-full border-[1.5px] font-['Outfit'] font-semibold text-xs tracking-[0.08em] uppercase transition-all hover:-translate-y-0.5"
  style={{ backgroundColor: '#F5E642', color: '#0D0D0D', border: 'none' }}
>
```

**Card Styling Pattern:**
```tsx
<Card className="border rounded-2xl transition-all hover:-translate-y-1" style={{ backgroundColor: '#C8D5B0', borderColor: '#E8E5E0' }}>
```

**Layout:** Tailwind utility classes for layout (`flex`, `grid`, `space-y-*`, `gap-*`, `p-*`, `max-w-7xl mx-auto`)

## Error Handling

**Patterns:**
- Minimal error handling throughout -- this is a prototype/demo application
- Image fallback pattern in `src/app/components/figma/ImageWithFallback.tsx`: uses `onError` callback to swap to a fallback SVG
- No try/catch blocks, no error boundaries, no global error handling

## Logging

**Framework:** None

**Patterns:**
- No logging statements exist in the codebase
- No console.log, no structured logging

## Comments

**When to Comment:**
- JSX section comments using `{/* Section Name */}` to delineate major layout areas
- Inline comments for non-obvious logic: `// Mock transcript generation`, `// Simulate AI processing`
- Comments in config files explain constraints: `// The React and Tailwind plugins are both required for Make`

**JSDoc/TSDoc:**
- Not used anywhere in the codebase

## Function Design

**Size:** Screen components are large monolithic files (100-600 lines). No decomposition into smaller files.

**Parameters:** Component props are destructured inline. State setter functions use the spread pattern: `setInsightWidget({ ...insightWidget, input: e.target.value })`

**Return Values:** Components return JSX. No custom hooks or utility functions that return complex types.

## Module Design

**Exports:**
- Screen components: single named export per file (`export function WelcomeScreen()`)
- UI components: multiple named exports at file bottom (`export { Card, CardHeader, CardContent }`)
- No default exports except `src/app/App.tsx`

**Barrel Files:** Not used. All imports reference specific files directly.

## State Management

**Approach:** Local component state only via `useState`. No global state management library.

**Pattern:** Each screen manages its own state. No shared state between screens. Navigation passes no state -- each screen has hardcoded mock data.

## Mock Data Pattern

All data is hardcoded directly in component files as local constants or initial state values. There are no API calls, no data fetching, no backend integration. Simulated async behavior uses `setTimeout`:
```tsx
setTimeout(() => {
  setInsightWidget({
    ...insightWidget,
    isProcessing: false,
    showResponse: true,
    response: "Based on your recent thoughts...",
  });
}, 2500);
```
